Build and Test Conflicts in The Wild

Abstract:

Context: : When collaborating, developers create and change software artifacts often without full awareness of changes being made by other team members. While such independence is essential for non small teams, and might increase development productivity, it might also result in conflicts when integrating developers changes. Goal: To better understand part of these conflicts__ the ones revealed by failures when building and testing integrated code__ in this paper we investigate their frequency, structure, and adopted resolution patterns. Methodology: We analyze merge scenarios (merge commits and the associated integrated changes) from more than 500 Java projects that use Travis CI for continuous integration. To identify build and test conflicts, we select merge scenarios from git repositories, parse the Travis logs generated when building and testing the commits in a scenario, and check whether the logged error messages are related to the integrated changes. Results: We find and classify more than 130 build and test conflicts, and their resolution patterns. We also find and analyze build and test failures caused by immediate post integration changes, which are often performed with the aim of fixing merge conflicts. Most build conflicts are caused by missing declarations removed or renamed by one developer but referenced by the changes of another developer. These conflicts are often resolved by removing the dangling reference. Conflicting changes break not only existing tests, but also new and modified tests. To resolve build conflicts developers often fix the integrated code, instead of completely discarding changes and conservatively restoring project state to a previous commit. Conclusion: Based on the catalogue of build conflict causes derived from our study, awareness tools could alert developers about the risk of a number of conflict situations they currently do not support. Program repair tools could benefit from the derived catalogue of build conflict resolution patterns, and the study infrastructure, to automatically fix conflicts. The scripts we developed for identifying the classes changed in a merge scenario, and involved in a test case failure, could be used to help developers to understand the failure and fix test conflicts.

Replication Study

During this work, we design a study involving three main steps as illustrated bellow. The first step mines project repositories in GitHub and Travis for getting the information required for the experiment. The second step parses the Travis build logs of the collected scenarios to filter out scenarios with build breakages caused by non integration related causes such as execution timeout of the Travis services. Finally, the last step confirms conflict occurrence, and classifies conflicts and the adopted resolution patterns.
