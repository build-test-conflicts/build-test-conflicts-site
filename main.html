Build and Test Conflicts in The Wild

Abstract:

Context: When collaborating, developers create and change software artifacts without being aware of other team members work. While such independence is essential for non small teams increasing development productivity, it might also result in conflicts when integrating contributions. Goal: To better understand part of these conflicts— those revealed by failures when building and testing integrated code— we investigate their frequency, structure, and adopted resolution patterns. Methodology: We analyze merge scenarios from more than 500 Java projects that use Travis CI. To identify build and test conflicts, we select merge scenarios from git repositories, parse the Travis logs generated when building and testing the commits in a scenario, and check whether the logged error messages are related to the integrated changes. Results: We find and classify more than 650 build and test conflicts, and their resolution patterns. Most build conflicts are caused by missing declarations removed or renamed by one developer but referenced by the changes of another developer. These conflicts are often resolved by removing the dangling reference. Conflicting changes break not only existing tests but also new and modified tests. We also identify build and test failures caused by immediate post integration changes. Conclusion: Based on the catalogue of build conflict causes derived, awareness tools could alert developers about the risk of a number of conflict situations. Program repair tools could benefit from the derived catalogue of build conflict resolution patterns, and the study infrastructure, to automatically fix conflicts. Developers could benefit from our scripts to understand test case failures and fix test conflicts.

Replication Study

During this work, we design a study involving three main steps as illustrated bellow. The first step mines project repositories in GitHub and Travis for getting the information required for the experiment. The second step parses the Travis build logs of the collected scenarios to filter out scenarios with build breakages caused by non integration related causes such as execution timeout of the Travis services. Finally, the last step confirms conflict occurrence, and classifies conflicts and the adopted resolution patterns.
